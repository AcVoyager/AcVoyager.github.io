关于Git的一些review

Posted on 2020-11-18 Edited on 2020-11-19 In 中文, English Views: Disqus:

最近由于一些group work导致对git的使用量激增。在使用中，也重新对之前半懂不懂的一些概念进行了复习。
本文便记载一些复习的内容。

Git的核心不外乎是其存储/数据结构，及其相互之间的关系。不论何时，官方文档永远是最好的资源:
What-is-Git

个人看来，重点有如下三处：

  1. Snapshots, Not Differences.

    也就是说，Git每次commit看似会显示这里修改了什么什么，那里添加/删除了什么什么，
    但是其存储的每个版本并不是这些变化(存储变化的VCs一般被称为Delta [$\Delta$] VCs)，而是存储一系列快照。具体存储的位置
    为Git directory的compressed database中。

  2. Three states.

    用Git管理的文件一般处于这三个状态中的一种: modified, staged, committed.

    • Modified: Modified means that you have changed the file but have not committed it to your database yet. 也就是说该文件被修改（内容增删改，文件本身的增删），但是并未committed。这种状态通常出现在已经被add到index区的文件
      发生改动的情况。如果是untracked的文件随你怎么改动，状态永远是untracked。
    • Staged: Staged means that you have marked a modified file in its current version to go into your next commit snapshot. 也就是刚才说的modified的文件被add到index区后，处于staged状态。staged说白了意思就是准备好
      要被committed了。
    • Commited: Committed means that the data is safely stored in your local database. 被commit的文件已经
      安全地保存到本地数据库中。

  3. Git存储数据结构

    祭出下面这张图，一定要bear in mind。记不住。。。请耗子尾汁😂

    • Working directory: The working tree is a single checkout of one version of the project. These files are pulled out of the compressed database in the Git directory and placed on disk for you to use or modify. 也有叫法是working tree，就是你当前编辑的这个版本基础，可能是之前的一次commit，也可能是
      init状态。你直接修改的就是这个区的文件。那么working directory的内容是从git的本地（压缩）数据库中抽取并放到硬盘上的，供你编辑。
    • Staging area: 经常说的index就指这个区域。index本身是一个文件，存储在Git directory内。作用就是保存你准备commit的改动。
    • Gir directory: The Git directory is where Git stores the metadata and object database for your project. git clone就是下载的这玩意儿，然后再从中抽取最新的commit记录变成你看到的working directory（逗号后面是我自己根据上面working dir的概念的理解哈）。每次commit之后，都把当前staging area中的内容存为一次快照存到本地（压缩）数据库中。

重点说完了，个人觉得记忆上述内容，比较易于理解的角度是从tracked file，也就是index区中记录的文件。我们从一个初始的刚刚git init的文件夹角度思考，一开始所有文件的状态是untracked。然后你自己编辑.gitignore，这里面提及到的文件会被彻底忽视，也就是连untracked状态都不会有因为你压根不想让git考虑它们（后文会讲一下.gitignore）。之后你做了些修改，然后你要git add。此后，你add涉及到的所有文件便成为tracked file。除非你在未来通过改变.gitignore或者git rm从Git Directory中删掉他们，否则这些tracked file之后只可能有两种状态：modified, or commited. 好了你现在满足了，commit交卷走人。那么Git directory的compressed database中便会创建一个包含这些tracked file在staging area中改动的快照并保存。你当前的working directory就是最新的这个commit。

关于.gitignore，语法层面没什么好说的，用正则表达式，中枪的文件被git忽略，也可以用！号来表示不要忽略某文件/目录，这种情况一般是父目录被忽略了但是里面有些资料你不想让git忽略。

值得一提的是，如果之前有commit记录，然后你想从现在这个commit记录开始修改.gitignore并令其生效。.gitignore被修改并不会自动地影响tracked file，所以这个时候你需要用比如git rm -r --cached <file>指令来将当前index（staging area）区中的指定删除掉，然后重新git add <file>再add入index区中。刚才那个git rm影响到的文件会被视为是staged的改动，附截图帮助理解：

可以看到，在git rm –cached以后，index区中的文件会被删除。实际上，这些文件被“降级”为untracked file. 具体见下图：

但是如果你不是–cached而是直接git rm删除的话，文件会从index和work dir中被删掉。

git rm

Remove files matching pathspec from the index, or from the working tree and the index. git rm will not remove a file from just your working directory. (There is no option to remove a file only from the working tree and yet keep it in the index; use /bin/rm if you want to do that.) The files being removed have to be identical to the tip of the branch, and no updates to their contents can be staged in the index, though that default behavior can be overridden with the -f option. When –cached is given, the staged content has to match either the tip of the branch or the file on disk, allowing the file to be removed from just the index.
